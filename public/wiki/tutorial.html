Goal

Build a private read-only bridge from an Obsidian vault to AI clients.

The bridge scans selected vault notes, creates a safe derived index, hosts that index behind authentication, and exposes read-only MCP tools so clients can search and fetch context without touching the vault filesystem.

The important design choice is separation: the live Obsidian vault remains the source of truth, while the hosted server only stores a filtered copy of selected content. That makes the remote system easier to reason about and easier to shut off or rebuild.

System Shape

1. Vault scanner

Runs locally against a vault copy. It reads Markdown files, rejects unsafe paths, parses metadata, redacts secrets, chunks notes, and creates index documents.

2. Hosted server

Runs on Vercel. It accepts admin syncs, stores the index, answers health checks, handles OAuth, and exposes MCP over HTTP.

3. AI clients

ChatGPT, Claude, Codex, or Inspector authenticate, call MCP tools, receive structured search results, and fetch only indexed documents.

The full request loop

  1. The local indexer scans the approved vault snapshot.
  2. The indexer posts a replacement index to /admin/sync.
  3. The server stores documents in JSON locally or Postgres in production.
  4. A client authenticates through OAuth or a development bearer token.
  5. The client calls /mcp with a JSON-RPC tool request.
  6. The MCP layer calls storage methods such as searchVault or fetch.
  7. The client receives structured results with path, snippet, score, citation URL, and Obsidian URI.

Data Model

If you understand the data model, the rest of the system gets easier. Everything revolves around VaultDocument.

FieldWhy it exists
idStable id for one indexed chunk. Search returns this id; fetch uses it.

metadata.source_policyStores evidence for why this document was allowed.

Source Policy

The source policy is the safety gate. It runs before a note becomes remotely searchable.

Deny first

Reject credentials, daily notes, finance, identity, legal, archives, review-gated notes, sensitive tags, and non-Markdown input before considering allow rules.

Allow narrowly

Allow the Task Hub, active project homes, and selected technical/reference folders. Unselected reference folders are denied by default.

The rebuild lesson is simple: write the allowlist in code, test it directly, and record the matched rule in metadata so every indexed chunk can explain why it exists.

Build The Indexer

1

List Markdown files

Walk the vault folder recursively, skip hidden folders and node_modules, keep only .md files, and sort the list for deterministic output.

2

Reject obvious paths before reading

Run path-only policy checks before opening files. That reduces accidental exposure because denied folders do not even need content parsing.

3

Read, redact, parse

Read Markdown, replace credential-like patterns with [REDACTED:name], parse frontmatter with gray-matter, and extract title, tags, status, headings, wikilinks, and tasks.

4

Apply full policy

Use path, tags, and status to decide whether the note is allowed. Denied notes increment stats and stop there.

5

Create chunks

Split the note by headings. Oversized sections are sliced into smaller chunks. Each chunk becomes one VaultDocument.

6

Emit the index

The finished index has generated_at, vault_root, documents, and stats. It can be written to JSON locally or synced to the server.

Build Storage

The server uses one interface with two implementations. That is why local tests and production behave the same way.

ImplementationWhen usedHow it works
JsonIndexStoreLocal development and testsKeeps documents in memory and writes a JSON snapshot to disk on sync.
PostgresIndexStoreProductionCreates tables for metadata, documents, OAuth clients, and token-use records. Sync runs as one transaction.

Both stores expose the same methods: replace, health, searchVault, searchNotes, searchSections, fetch, fetchByPath, listNotes, recentNotes, activeProjects, indexStatus, and debugSearch.

Expose MCP Tools

MCP tools are the public API for AI clients. The server currently exposes ten read-only tools.

Discovery

list_notes, recent_notes, and active_projects let a client browse known safe notes without guessing paths.

Search

search, search_notes, and search_sections return note-level or section-level matches with snippets and match reasons.

Fetch

fetch and fetch_note_by_path return indexed text only when the id or path already exists in the allowlisted index.

Diagnostics

get_index_status and debug_search explain freshness, scope boundaries, and why a query may not find results.

Add OAuth

OAuth exists so real clients do not need a shared static password as their long-term API token. The self-hosted OAuth server supports dynamic registration, authorization code + PKCE, refresh tokens, and replay protection.

  1. The client discovers metadata from /.well-known/oauth-authorization-server.
  2. The client registers redirect URIs at /oauth/register.
  3. The human owner authorizes through /oauth/authorize with the connector password.
  4. The server returns an authorization code to the client redirect URI.
  5. The client exchanges the code and PKCE verifier at /oauth/token.
  6. The server signs a one-hour access token and a refresh token.
  7. Authorization codes and refresh tokens are consumed once by recording their JWT ids.

Deploy It

The deployed shape is Vercel for the Express Function and Neon Postgres for the index plus OAuth persistence.

  1. Create/link the Vercel project.
  2. Provision Postgres and set DATABASE_URL.
  3. Set PUBLIC_BASE_URL, MCP_SYNC_TOKEN, OAuth issuer/audience/secret/password/scopes, and allowed origins.
  4. Deploy with vercel deploy --prod -y.
  5. Run the remote OAuth smoke test.
  6. Connect clients to https://vault-mcp-connector.vercel.app/mcp.

What To Build Next

The read-only V1 server is intentionally conservative. The long-term read/write version should be designed as a separate layer with scoped write operations, approval UI, audit logs, conflict checks, backups, and rollback. Do not turn this V1 read-only server into a raw remote filesystem editor.
