Plain-English Overview

What this project does

It lets approved AI tools discover, search, diagnose, and fetch selected notes from an Obsidian vault. It is read-only. It does not edit notes. It does not expose the whole vault. It serves a controlled index that was built from allowlisted notes.

What this project does not do

It does not let AI browse the filesystem. It does not read the live vault on every request. It does not allow guessed note paths. It does not expose raw daily notes, credentials, finance, legal, identity, or archived sensitive material.

The simplest useful sentence

A local indexer scans only approved Markdown files, sends approved chunks to a hosted server, and the hosted server exposes read-only MCP tools for note discovery, section search, active project lookup, diagnostics, and controlled fetches.

Mental Model

Think of the hosted index as a photocopied binder, not the vault itself.

The Obsidian vault is the source of truth. The server only receives a derived, rebuildable, filtered index.

Why this design is safer

  • The hosted server does not need direct filesystem access to the vault.
  • The source policy runs before data is uploaded.
  • Denied notes never become remotely fetchable.
  • Clients can only call read-only tools; no edit, write, delete, or filesystem-browse tools exist.
  • Every fetched item has a private citation URL and metadata showing where it came from.

Request Flow

Indexing flow

  1. You run the indexer and point it at the vault copy.
  2. The indexer finds Markdown files.
  3. Each file path is checked against quick deny rules before reading content.
  4. Allowed candidate files are parsed for frontmatter, tags, status, title, links, and tasks.
  5. The full source policy decides whether the note is allowed.
  6. Allowed notes are split into heading-based chunks.
  7. Each chunk gets a stable id, title, text, URL, and metadata.
  8. The finished index is written locally or posted to /admin/sync.

Client request flow

  1. ChatGPT, Claude, Codex, or MCP Inspector calls /mcp.
  2. The server checks the request origin if an Origin header exists.
  3. The server checks the bearer token or OAuth token.
  4. The MCP SDK receives the JSON-RPC request.
  5. If the client asks for search, the store searches the index.
  6. If the client asks for fetch, the store fetches by id.
  7. If no document matches the id, the tool returns an error instead of data.

Glossary

MCPModel Context Protocol. A standard way for AI clients to call external tools.
ToolA callable function exposed to an AI client. This server exposes ten read-only tools for search, listing, recent notes, active projects, fetches, index status, and diagnostics.
OAuthA protocol for authorizing clients without sharing long-lived passwords with them.
PKCEA safety extension for OAuth authorization-code flows used by public clients.
JWTA signed token format. This server signs/verifies access, refresh, and authorization-code tokens.
JTIA token id. The server records used JTIs to prevent replay.
CORSBrowser origin protection. It controls which websites may call this server from a browser.
AllowlistRules that say what can enter the remote index.
DenylistRules that say what must never enter the remote index.
